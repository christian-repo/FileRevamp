    Create src/FileRevamp/Core/RenameOrchestrator.cs:
      Namespace: FileRevamp.Core
      Class RenameOrchestrator
        Constructor: RenameOrchestrator(IFileSystem fileSystem)
        Public method: IEnumerable<RenameResult> Execute(string directoryPath, WildcardPatternMatcher patternMatcher, bool dryRun)
          Pipeline per file:
            1. fileSystem.GetFiles(directoryPath, "*") → string[] filePaths
            2. For each filePath:
               a. filename = fileSystem.GetFileName(filePath)
               b. newFilename = patternMatcher.ApplyRemoves(filename)
               c. if newFilename is null → yield RenameResult.SkippedResult(filename, "No pattern matched")
               d. if newFilename == filename → yield RenameResult.SkippedResult(filename, "Transform produced no change")
               e. destPath = fileSystem.Combine(directoryPath, newFilename)
               f. if fileSystem.FileExists(destPath) → yield RenameResult.SkippedResult(filename, "Destination already exists (conflict resolution coming in Phase 2)")
               g. if dryRun → yield RenameResult.DryRunResult(filename, newFilename)
               h. else:
                  try { fileSystem.MoveFile(filePath, destPath); yield RenameResult.RenamedResult(filename, newFilename) }
                  catch (Exception ex) { yield RenameResult.FailedResult(filename, ex.Message) }

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI args → RenameSettings | User-supplied directory paths and patterns arrive untrusted; Spectre.Console.Cli parses them before Execute sees them |
| RenameOrchestrator → IFileSystem | Computed destination paths passed to MoveFile may be crafted to traverse outside the target directory |
| Pattern → Regex compilation | User wildcard pattern becomes a compiled Regex; catastrophic backtracking is possible with adversarial inputs |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | RenameOrchestrator.Execute — destPath construction | mitigate | After computing destPath, verify Path.GetDirectoryName(destPath) == directoryPath (normalized, absolute). Reject any path where the destination escapes the source directory — emit RenameResult.FailedResult with "Path traversal rejected". |
| T-01-02 | Tampering | RenameSettings.Path argument | mitigate | Resolve to absolute path with Path.GetFullPath at command entry; reject paths containing ".." segments after resolution if they leave the user's stated directory |
| T-01-03 | Denial of Service | WildcardCompiler.ToRegex — RegexOptions.Compiled on adversarial patterns | accept | Inputs are developer-supplied patterns, not end-user HTTP inputs; tool has no network surface; ReDoS risk is low |
| T-01-04 | Information Disclosure | Per-file output lines written to console | accept | Tool is designed to output filenames; no PII in filenames expected for this use case; developer tool with no multi-user surface |
| T-01-05 | Elevation of Privilege | IFileSystem.MoveFile targeting system files (e.g., C:\Windows\system32\) | mitigate | GetFiles scans only the user-supplied directory (non-recursive in Phase 1); no mechanism to target files outside the scan directory. Document in --help: "FileRevamp only renames files in the specified directory." |
| T-01-SC | Tampering | dotnet restore / NuGet package install | mitigate | Package Legitimacy: Spectre.Console.Cli 0.55.0 (spectreconsole.net, confirmed HIGH confidence), Microsoft.Extensions.FileSystemGlobbing 9.* (official Microsoft package), xUnit 2.9.x (xunit.net, official), FluentAssertions 7.x (official). All packages are [VERIFIED] from official sources per STACK.md research. No [ASSUMED] or [SUS] packages. |
</threat_model>
